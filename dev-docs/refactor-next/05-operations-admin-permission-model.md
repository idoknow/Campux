# 运营管理员权限模型

本文档记录 Campux 面向开源分发和官方托管服务时的账号/管理链路设计。

## 目标

官方服务会允许运营者直接创建校园墙并负责后续运营。因此系统不能只有 `system_operator` 这一种全局管理身份，否则所有运营管理都会被迫暴露在系统运维面板里。

新的角色边界：

| 身份 | 范围 | 能力 |
| --- | --- | --- |
| `system_operator` 系统运维 | 整个实例 | 管理所有租户、全局账号、平台身份、审计日志、队列和生命周期 |
| `operations_admin` 运营管理员 | 自己负责的校园墙集合 | 创建校园墙；查看和管理自己作为租户管理员加入的校园墙 |
| `TenantMembership.admin` 租户管理员 | 单个校园墙 | 管理本墙用户、封禁、元数据、机器人、发布目标和统计 |
| `TenantMembership.reviewer` 审核员 | 单个校园墙 | 审核稿件和查看统计 |
| `TenantMembership.submitter` 用户 | 单个校园墙 | 投稿、查看自己的稿件、取消待审核投稿 |

## 入口规则

登录后：

1. 普通账号只进入自己所属的校园墙。
2. 多墙账号进入校园墙选择页。
3. `system_operator` 在选择页看到“系统运维”入口。
4. `operations_admin` 在选择页看到“运营管理”入口。
5. 绑定 host 的访问仍然优先生效：如果当前域名对应某个校园墙，用户只能进入该校园墙。

## 运营管理员可见范围

运营管理员进入 `/ops` 后使用同一套运维接口，但服务端必须按其租户管理员 membership 收窄：

- 租户列表：只返回 `TenantMembership.role = admin` 的校园墙。
- 用户列表：只返回这些校园墙内的成员；不泄露全局系统角色。
- 审计日志：只返回这些校园墙相关日志。
- 发布队列：只返回这些校园墙相关的 `PublishAttempt` 统计。
- Bot/发布目标：只返回这些校园墙的 Bot 和发布目标。
- 生命周期修改：只能修改这些校园墙。

如果运营管理员创建新校园墙，创建者会被自动加入为该租户管理员。

## 系统运维保留能力

系统运维仍然是唯一可以：

- 查看所有租户和全局用户。
- 授予 `operations_admin` 或 `system_operator` 平台身份。
- 跨租户查看审计日志和队列。
- 管理没有归属到自己的校园墙。

系统运维创建租户时，所有系统运维会自动加入为租户管理员，避免新租户无人可进入。

## UI 原则

- “系统运维”文案只用于 `system_operator`。
- “运营管理”文案用于 `operations_admin`。
- 运营管理员面板不展示“全局用户”等全局措辞，改为“墙内用户”或“可管理用户”。
- 运营管理员的身份分配对话框不展示平台身份选项，只能添加/更新租户内身份。
- 普通租户内管理页仍然只负责单个校园墙的运营配置，不承载全局生命周期管理。
